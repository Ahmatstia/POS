05 CODING STYLE & ARCHITECTURE RULES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
ARCHITECTURE: Clean Architecture, feature-first folder structure
  lib/
  ├── core/
  │   ├── database/       ← AppDatabase, tables/, daos/, migrations/
  │   ├── design/         ← AppColors, AppTextStyles, AppSpacing (static const)
  │   ├── widgets/        ← AppButton, AppCard, AppTextField, StatusBadge, ShimmerBox
  │   ├── extensions/     ← BuildContext, String, int, DateTime extensions
  │   ├── utils/          ← formatCurrency(), formatDate(), validators
  │   └── constants/      ← ApiEndpoints, StorageKeys, AppConfig
  └── features/[feature]/
      ├── data/           ← repository_impl, datasource, model (freezed)
      ├── domain/         ← repository (abstract), entity, usecase
      └── presentation/   ← screen, widget, provider (riverpod)

DART CODE RULES (non-negotiable):
  - Every Widget class: single responsibility, max ~100 lines. Extract sub-widgets.
  - No business logic inside build() method. Ever.
  - No inline styling magic numbers. All spacing from AppSpacing.*, colors from AppColors.*
  - Riverpod: use @riverpod annotation + code generation. No manual Provider() calls.
  - Every async call: must handle loading / data / error state explicitly (AsyncValue)
  - Monetary values: always stored and calculated as int (Rupiah, no decimals, ever)
  - No String for status — use sealed class or enum with extension methods
  - All models: immutable, generated with freezed + json_serializable
  - Repository pattern: domain layer never imports dart:io or package:http
  - File naming: snake_case. Class naming: PascalCase. No abbreviations (no "Mgr", "Util")
  - Every public method/class: dartdoc comment (///) — brief, describes intent not code
  - No TODO comments in generated code — either implement or leave a GitHub issue

WIDGET CODE PATTERN:
  class ProductCard extends StatelessWidget {
    const ProductCard({super.key, required this.product, this.onTap});
    final Product product;
    final VoidCallback? onTap;

    @override
    Widget build(BuildContext context) {
      return GestureDetector(
        onTap: onTap,
        child: _CardContent(product: product),  // always extract to private sub-widget
      );
    }
  }

06 STATE FLOW RULES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CART STATE (core of POS):
  - Cart is a single Riverpod StateNotifier — single source of truth
  - Cart items: Map — O(1) lookup, no duplicates
  - Every mutation returns new immutable state (copyWith). No direct mutation.
  - Cart subtotal, tax, discount: computed via selectors (not stored in state)
  - CartItem quantity: min = 1, enforced at state level (not UI)
  - Hold orders: List in separate provider, max 10 concurrent holds

SCREEN STATE RULES:
  - Every screen: has exactly 3 states → loading / content / error
  - Loading: full-area shimmer skeleton, not partial spinners scattered around
  - Error: branded error screen with retry button + error code (not raw exception string)
  - Empty: branded empty state — never show a blank screen

NAVIGATION STATE:
  - GoRouter with typed routes (go_router_builder code gen)
  - Deep-link support for every screen (for manager override flows)
  - Route guards: auth check, role-based, shift-open check
  - Never use Navigator.push directly — always go through router

FORM STATE:
  - Use flutter_form_builder or reactive_forms — never raw TextEditingController soup
  - Validate on submit + show inline field errors (not SnackBar)
  - Dirty state tracking: warn user on navigation away from unsaved form

07 MICROINTERACTION GUIDANCE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
REQUIRED ANIMATIONS (use flutter_animate package):

  Product added to cart:
    → Card: scale 0.95 → 1.05 → 1.0 over 180ms (spring curve)
    → Cart badge: bump scale 1.0 → 1.3 → 1.0 over 200ms
    → Cart panel: new row slides in from right + fades in (150ms)

  Payment success:
    → Full-screen overlay: checkmark draws itself (stroke animation, 400ms)
    → Background: brief pulse of emerald-50 tint
    → Receipt card: slides up from bottom (300ms, ease-out)

  Number pad input:
    → Each key: scale 0.94 on press → 1.0 on release (80ms)
    → Amount display: slide-up replace (not blink) when digit added

  Category filter tap:
    → Active pill: width-animated underline slides to new tab (200ms)
    → Grid: staggered fade-in + translateY(+8px → 0) per card, 30ms delay between items

  Hold order:
    → Cart slides out to the left (300ms), new empty cart slides in from right

  Error shake:
    → Input field or card: horizontal shake (±4px, 3 cycles, 300ms total)

HAPTIC FEEDBACK:
  Item add to cart    : HapticFeedback.lightImpact()
  Payment success     : HapticFeedback.heavyImpact()
  Destructive action  : HapticFeedback.mediumImpact()
  Error / warning     : HapticFeedback.vibrate()

LOADING SKELETONS:
  → Match exact layout of real content (not generic rectangles)
  → Product grid skeleton: 4×3 cards with correct 3:4 aspect ratio
  → Transaction row skeleton: 3 lines matching real row height

08 PERFORMANCE CONSTRAINTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
HARD PERFORMANCE TARGETS:
  Cold start to POS screen         : < 2.5 seconds
  Add item to cart (UI response)   : < 16ms (single frame, 60fps)
  Product grid scroll              : 60fps, zero jank
  Receipt print trigger to done    : < 3 seconds
  API response → UI update         : < 300ms perceived

RENDERING RULES:
  → Product images: use cached_network_image with memCacheWidth/Height set
  → Long lists: always use ListView.builder / SliverList — never ListView(children:[])
  → Product grid: use SliverGrid with delegate — not GridView.count
  → Never rebuild entire screen — use Consumer or select() to scope rebuilds
  → Expensive computations (totals, report aggregation): run via compute() in Isolate
  → Image assets: WebP format, max 80KB per product image

OFFLINE-FIRST RULES:
  → POS core (scan, add to cart, checkout with cash): must work 100% offline
  → Sync queue: failed API calls → stored in Drift sync_queue table →
    retry on reconnect with exponential backoff (1s, 2s, 4s, 8s, max 5 retries)
  → Conflict resolution: last-write-wins for product data; server-wins for inventory stock
  → Always show connectivity status indicator in top-right corner (1 line, non-intrusive)
  → On reconnect: process sync_queue oldest-first, mark status = synced on success

DRIFT-SPECIFIC PERFORMANCE:
  → All DAOs use .watch() streams — never poll or re-query on timer
  → Expensive JOIN queries: use computed columns + indexes on foreign keys
  → Enable WAL mode: PRAGMA journal_mode = WAL (set in beforeOpen)
  → Enable foreign keys: PRAGMA foreign_keys = ON (set in beforeOpen)
  → Use database transactions for multi-table writes (e.g. checkout: insert
    transaction + transaction_items + decrement stock in one atomic transaction)

11 DRIFT DATABASE SCHEMA
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
DATABASE FILE: lib/core/database/app_database.dart

GLOBAL RULES FOR ALL TABLES:
  - id          : INTEGER PRIMARY KEY AUTOINCREMENT
  - createdAt   : DateTime NOT NULL
  - updatedAt   : DateTime NOT NULL
  - isDeleted   : BOOLEAN DEFAULT false   (soft delete — never hard delete data)
  - Enums       : stored as INTEGER, mapped via IntEnum type converters
  - Foreign keys: always declared with references(), cascade on delete where appropriate
  - All prices/amounts: INTEGER in Rupiah — no REAL, no double, ever

DATABASE CLASS:
  @DriftDatabase(tables: [
    Users, Categories, Products, ProductVariants,
    Customers, Shifts, Transactions, TransactionItems,
    StockMovements, SyncQueue,
  ])
  class AppDatabase extends _$AppDatabase {
    AppDatabase() : super(_openConnection());

    @override
    int get schemaVersion => 1;  // increment on EVERY schema change

    @override
    MigrationStrategy get migration => MigrationStrategy(
      onCreate: (m) async {
        await m.createAll();
        await _seedDefaultData();  // insert default categories + owner user
      },
      onUpgrade: (m, from, to) async {
        for (var step = from; step < to; step++) {
          await _runMigration(m, step);
        }
      },
      beforeOpen: (details) async {
        await customStatement('PRAGMA foreign_keys = ON');
        await customStatement('PRAGMA journal_mode = WAL');
      },
    );
  }

DAO PATTERN:
  @DriftAccessor(tables: [Products, Categories, StockMovements])
  class ProductDao extends DatabaseAccessor
      with _$ProductDaoMixin {
    ProductDao(super.db);

    /// Watch all active products with category, ordered by name.
    Stream> watchAllActive() {
      return (select(products)
        ..where((p) => p.isDeleted.equals(false) & p.isActive.equals(true))
        ..orderBy([(p) => OrderingTerm.asc(p.name)])
      ).join([
        leftOuterJoin(categories, categories.id.equalsExp(products.categoryId)),
      ]).watch().map((rows) => rows.map(ProductWithCategory.fromRow).toList());
    }

    /// Search by name or barcode — used by POS search bar + scanner.
    Future> search(String query) {
      return (select(products)
        ..where((p) =>
          p.isDeleted.equals(false) &
          (p.name.like('%$query%') | p.barcode.like('%$query%')))
      ).get();
    }

    /// Decrement stock atomically. ALWAYS use this — never update stock directly.
    Future decrementStock(int productId, int qty) {
      return transaction(() async {
        final p = await (select(products)
          ..where((t) => t.id.equals(productId))).getSingle();
        await (update(products)..where((t) => t.id.equals(productId)))
          .write(ProductsCompanion(
            currentStock: Value(p.currentStock - qty),
            updatedAt: Value(DateTime.now()),
          ));
      });
    }
  }

MIGRATION RULES (non-negotiable):
  ✓ NEVER drop a column — add nullable column or use DEFAULT instead
  ✓ NEVER rename a column — add new, migrate data, keep old as deprecated
  ✓ Every schema change = schemaVersion++ = new migration file
  ✓ Test migration from version N to N+1 with real data before shipping
  ✓ Use SchemaVerifier in integration tests to validate every migration
  ✓ Write migrations using customStatement() in raw SQL for clarity
  ✗ NEVER use m.recreateAllViews() in production — destructive
  ✗ NEVER change a column type — add a new column instead
